1、动态加载对象

1）分析

通过ClassLoader类进行加载

通过调试发现，loadclass会进入Launcher然后再调用到当前类的一个重载方法，然后去加载这个类，这里就用到了双亲委派机制，一级一级往下看能不能加载：BootstrapClassLoader->ExtensionClassLoader->ApplicationClassLoader -> customClassLoader（通过不同的路径去加载，对应的加载器加载对应路径的class），避免触发安全漏洞。

然后就会进入到URLClassLoader中的findClass方法，然后通过调用进入到SecureClassLoader中的defineClass，最终会进入到ClassLoader类中的方法defineClass，它的参数：类名，类的字节码，类的起始位置，类的长度，保护域（一般为null），接着进入当前类的native方法。

继承关系：ClassLoader->SecureClassLoader->URLClassLoader->(ExtClassLoader、AppClassLoader)

既然底层方法是ClassLoader中的defineClass，那么我们可以使用反射调用加载任意的类。

2）ClassLoader利用

攻击类：

利用ClassLoader动态加载对应的类，使用底层调用的defineClass方法。

3）URLClassLoader利用

攻击类还是一样

利用URLClassLoader，使用其loadClass方法（父类的方法），也可以使用findClass（使用反射）

4）UnSafe利用

这个类使用了单例模式，需要通过反射创建，里面有一个字段theUnsafe就是那个单例。

5）总结

  • URLClassLoader#loadClass -> 任意类加载（file/http/jar）
  • ClassLoader#defineClass -> 字节码加载任意类 （私有方法）
  • Unsafe#defineClass -> 字节码加载任意类 （类单例）

2）CC1链

漏洞利用版本

  • Commons Collections<=3.2.1
  • jdk8版本在71之前

CC链最主要的类就是org.apache.commons.collections.Transformer，它是一个接口，可以理解为是一个转换器，它只有一个方法，将输入的对象进行转换。

1、接口实现类

它一共有14个实现类。其中使用较多的：

org.apache.commons.collections.functors.ChainedTransformer这是一个链式转换器，它的接口实现方法就是将输入的对象以链式进行转换，构造方法需要传递一个数据类型为Transformer的实现类。

org.apache.commons.collections.functors.ConstantTransformer这是个常量转换器，构造时需要给一个常量，然后接口方法无论输入什么都返回它。

org.apache.commons.collections.functors.InvokerTransformer这是一个任意方法调用的转换器，调用输入对象的对应方法，只能是公有的。

org.apache.commons.collections.functors.NOPTransformer这是转换器无论输入什么都返回你的输入。。

org.apache.commons.collections.functors.InstantiateTransformer这个调用任意构造方法的转换器，通过输入的参数等进行实例化。

这条CC1链后面的链式都一样，只不过用LazyMap替换了TransformedMap，LazyMap调用的是get方法。get方法的调用者实在太多了，我们直接利用现成的payload去复现吧。

1、分析

LazyMap构造方法也是受保护的，它的创建和TransformedMap类似，通过静态方法创建，我们只需要调用其get方法就可以调用到后面的链，并且factory还是用户提供的，所以利用非常简单，只需要保证在map中没有这个get方法参数的key就行了，他就能走进if，一种单例懒加载的实现。

上一层的调用为：annotationInvocationHandler.invoke()，我们知道这个类是动态代理的一个调用处理器类，invoke方法就是去调用代理方法的。所以我们只需要创建代理类，调用其方法就能去执行invoke。而它这里使用的代理类是一个Map，入口类还是annotationInvocationHandler类，调用它的readObject方法触发其代理的invoke。

因为Invoke做了处理，所以观察其处理：

首先可以看到，memberValues是我们可以控制的，所以就利用这个函数。

然后看61行的第一个判断，如果方法名为equals，参数为1并且参数为Object类型就退出。

64行判断，如果参数个数不为空就退出，因为它是来代理注解的，注解方法都是没有参数的。

然后就去switch判断，如果是toString、hashCode、annotaionType就退出，其他都能走到get方法。

所以我们通过代理对象调用方法时就得调用一个无参方法就可以完成整条链，发现在readObject方法中调用了map的一个无参方法。

思路：通过代理这个map，当它调用了entrySet方法时，就去进入到annotationInvocationHandler类中的invoke，然后invoke就会去调用LazyMap的get方法，此时整条链就连起来了，只能说非常巧妙。

这个时候还需要理清，需要被反序列化的annotationInvocationHandler（这里记为a1），它的memberValues是一个代理的map，当我们调用它的entrySet方法就会触发代理的invoke方法，然后代理的类还是annotationInvocationHandler（记为a2）这个类，它通过invoke就会去调用get方法，此时a2里面的memberValues就是LazyMap，然后里面通过transform去调用整条链。

1、URL链

了解CC6链之前，先了解一下URL链。

URL链一般都用作poc，只需要观察是否有请求即可。

URL链的核心是URL+Map。

HashMap

在HashMap中的readObject方法会将键值对反序列化过来，并且key会的hash会重新计算，调用hash方法。

而这个hash方法的实现会调用这个对象的hashCode方法。

URL

在URL类中，它的hashCode方法是先判断，然后调用URLStreamHandler这个类的hashCode方法。

而这个类在调用hashcode方法时，获取请求一下该地址。

并且URL类还是可以序列化的，此时一条链就诞生了。

链实现

7）CC2链
