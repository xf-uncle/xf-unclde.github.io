shiro550分析

1、简介

简介

Shiro-550反序列化漏洞大约在2016年就被披露了，成为近几年最好利用的RCE漏洞的原因：Shiro框架使用广泛，漏洞影响范围广；攻击payload经过AES加密，所以很多安全防护设备无法识别/拦截攻击等

环境搭建

然后就配置一下tomcat即可运行了。

2、漏洞分析

首先进入主界面，然后打开bp抓包，点击到登录页面，选用上面的用户进行登录，并且选中remember me选项。

登录之后，会调转到主页，这个时候发现请求包中存在着一个字段叫rememberMe，而且它的值非常的长

要知道一般的cookie都会很短，猜测这里面可能保存着一些信息，一般都是一些序列化的值，我们打开idea去搜索类，找到了org.apache.shiro.web.mgt.CookieRememberMeManager这个类，通过类名可以获取它是对这个cookie值进行管理的一个类。

通过查找发现有一个方法getRememberedSerializedIdentity，它的名字就是函数的作用，用于获取rememberMe序列化的信息。

发现通过cookie读取之后，将其进行了base64的解密，我们查看这个方法的上层调用者。找到了org.apache.shiro.mgt.AbstractRememberMeManager类中的getRememberedPrincipals方法。

发现它在获取了rememberMe的值之后又调用了这个函数，将字节数组转换为主体，应该就是完整的信息。

通过跟进方法发现方法调用，获取到了：它将字节数组进行解密，然后调用了反序列化的方法，那就与我们猜测的一样。

虽然我们知道了它是调用反序列化，但是我们如果不知道如何构成密文还是没办法，所以跟进加密方法观察，发现它先获取了一个加密服务，然后通过该服务进行解密，看到了参数是密文以及一个方法：获取用来解密的加密密钥，也就是这是一个对称加密，我们跟进这个获取密钥的方法。

发现它返回的是一个成员变量，我们查看是否有函数给这个成员变量赋了值。

然后找到了setDecryptionCipherKey -> setCipherKey -> AbstractRememberMeManager这个调用链

发现它在构造函数里面就给了它赋值，赋的值还是一个常量。

成功获取到了key，发现它是经过AES加密然后使用Base64编码的。既然有了key的构成方法，并且key还是一个固定的值，我们就可以创建一个反序列化链了。

3、漏洞检测

既然有了漏洞利用的想法，我们就要想攻击它的哪一个库，虽然shiro项目默认是带CC的，但是它的scope是test，也就是在项目测试运行时是有这个包的，但是在打包发布时是不会去打包它的，也就是我们无法构成CC链攻击。

那既然这样，我们只能使用URL那条链去检测一下。

先将其进行序列化，然后通过AES进行加密，AES加密的模式为CBC，还需要处理向量，它的值在decrypt方法中有写到：

就是将密文的前多少个字节当做处理向量，所以我们的iv可以随便取。

使用步骤：使用bp生成一个dns，然后再将其放入URL链中进行反序列化，将反序列化的值进行aes加密以及base64编码，接着将rememberMe的值修改为对应生成的值，发送请求包即可造成注入！

注意：需要修改或者删除JSESSIONID，因为这个值是表示着当前会话，会话中是存有内容的，它的值没变是不会去检测rememberMe的值。

发现响应里有rememberMe等于deleteMe这个值。

通过调试发现解密之后会调用org.apache.shiro.io.DefaultSerializer类中的反序列化方法。

4、利用CC链

因为没有CC依赖，所以我们导入CC依赖，版本<=3.2.1都行。

然后使用CC6链获取反序列化的值，再进行AES加密，然后修改rememberMe的值，发包查看。

虽然请求成功了，但是服务器没有任何反应，查看日志信息，发现有报错。

它说无法加载Transformer数组。

通过调试发现它的反序列化方法，并不是原生的readObject，而是重新构造了一个对象，进行封装了一层。

跟进去看，发现它重写了resolveClass方法，这个方法就是classLoader在加载类的时候会去判断这个类是否被链接过，如果链接过就只做一个简单返回，否则就会根据java规范去链接，当我们重写它时，就会调用我们的重写方法去链接。

它父类的默认操作就是做了一个类加载。而它重写之后是调用shiro封装的类加载。

通过源码发现，它会调用当前线程的类加载器、当前的类加载器以及系统/APP类加载器去加载那个类，最多3次，无法加载就抛出异常，加载成功就返回。

我们知道我们使用tomcat打包后的外部库都是放在WEB-INF\lib中的，所以第三个加载方式加载不到，这里前两处用的都是同一个加载器。最后都会调用到tomcat针对webapp的类加载器，得到的结论：

  • 在WEB-INF\lib下面的类会调用webAppClassLoader，它无法加载数组
    • 最终都是调用了URLClassLoader进行加载，而它的加载逻辑是把类名中的.替换为\然后拼接.class，但是如果我们要加载数组类，名字处理后肯定是加载不到的，因为数组类名字前面会有[
  • 不在WEB-INF\lib下面的类会调用Class.forname加载，可以加载数组

也就是shiro只能加载不在WEB-INF\lib下面的数组类。

观察CC链的调用图：

通过链的总结图可以发现，如果不用数组就得不去使用chainedTransformer，所以就得结合CC2链进行调用。

成功利用shiro反序列化漏洞与CC6+CC2链结合攻击。

我们知道在Shiro原生下是没有CC依赖的，而Shiro中是有commons-beanutils这个依赖的，它是存在着反序列化漏洞的。

这个CB（commons-beanutils）依赖是对javabean的增强类库。

通过获取的这些信息，想到了之前在CC3链中，有一个对象为templatesImpl，它里面关键的函数就是newTransformer，而这个函数有一个调用链就是当前类中的getOutputProperties，并且这个类也不是特殊类型，所以还是会用getSimpleProperty方法。

那我们就可以让CB获取这个beanName，然后就会调用它的getOutputProperties方法，接着就是我们的CC3链了

能够成功执行链！接着我们就需要找到getProperty它的上层调用者，找到了org.apache.commons.beanutils.BeanToPropertyValueTransformer#transform方法，但是它没有实现可序列化接口，无法利用。

但是我们发现了另一个关键点：compare方法，我们知道之前在CC4那条链中就是利用这个方法的，虽然我们没有CC4的版本依赖，但是我们可以利用priorityQueue，它是jdk自带的类，通过它的readObject方法调用compare构建反序列化链，并且他也是一个Comparator，很好利用。

3）构造调用链

BeanComparator调用PropertyUtils.getProperty需要一个property，刚好我们可以控制，它的值为outputProperties。

只需要在构造函数传递它即可。

这里放入tc的原因：

如果不放入tc，PriorityQueue默认就没有comparator，那么在add元素时

就会进入到siftUpComparable方法，这个x是add的元素，它会将x转换为Comparable，但我们知道我们增加的元素不是它的子类，所以会造成转换异常。

如果放入bc，那么在调用时就会走上面一个方法，之前在CC4链的时候遇到过，就是放入元素会调用它的compare方法，导致我们在序列化时就会造成链的执行，所以我们这里放入一个临时的tc，它并不会影响class文件的内容，就是为了防止报错，这也是构造exp的一种思路，填充一个无意义的数据然后再将其修改，最终序列化的结果就只是对象的信息而已，跟处理的流程无关。
